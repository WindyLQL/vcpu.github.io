openstack SDN vlan多外网fip环路问题解决

发表于 2018-02-26 | 分类于 网络 | | 阅读次数

问题描述

组网环境基本介绍：

br-int是ovs交换机，qg和qr是router的两个接口，qg连接外网，qr连接内网vcp网络

云主机位于计算节点2的br-int上

leaf1、leaf2、spine、tor为物理交换机，其中外网的网关在spine上

现象描述

配置vlan外网，申请fip，最后将fip和云主机绑定后，整个openstack网络环境出现大量的新绑定的免费arp数据包，网络瘫痪。从tor交换机无法连接到网络节点和计算节点的管理地址了。

在qg、en02、网络节点bond0和计算节点bond0均可以抓去到arp大量相同的广播数据包

tcpdump抓取数据包如下：

大量的192.168.209.167免费arp情况。

初步分析

fip的免费arp广播报文在vlan209广播域内出现环路

环路可能性分析

环路可能性1:

环路描述：环路方向入上红色虚线所指，发生在网络节点上的vlan209广播域内，环路方向从br-public的eno2 到qg到bond0 最后到交换机1

验证：网络节点bond0口不允许tag209，查看qg和bond0是否可以抓去到数据包，如果可以抓到数据包，代表此推理正确

验证结果：eno2可以抓取到fip免费arp广播数据包，但是qg和bond0均不能抓取到fip免费arp广播数据包

结论：此种可能性不成立

环路可能性2:

环路方向：交换机1 -> 网络节点bond0 -> 虚拟交换机br-int -> 虚拟交换机br-public -> eno2

验证：配置交换机1和交换机2 连接网络节点bond0 口，允许tag209通过，且配置交换机1 连接eno2口交换机口不允许tag209

验证结果：在qg 和 eno2 bond0可以抓取到数据包，此环路成立

精细验证过程

因为数据包环路路径为，交换机1 -> 网络节点bond0 -> 虚拟交换机br-int -> 虚拟交换机br-public -> eno2

那么为什么数据包从bond0过来后经过虚拟机ovs交换机br-int可以转发给qg和eno2 ？

我们尝试产生相同arp广播数据包进行尝试：

ovs-appctl ofproto/trace br-int in_port=22,dl_src=fa:16:3e:e8:a3:36,dl_dst=ff:ff:ff:ff:ff:ff,arp,arp_spa=192.168.209.167,dl_vlan=209 -generate

发现免费arp广播数据包最后交给ovs 虚拟机交换机br-int的控制器处理了，于是查看代码，发现，该控制器会查找回复该arp，如果没有查找到后，会进vlan范围内进行广播，我们这种情况就属于后者

因此qg 和 br-public 上的eno2可以抓取到该广播数据包，这也就是环路产生的根本原因了。

解决办法

因流量特性，具体不多说了，可在交换机连接网络节点bond0口上不允许tag209通过。或者修改控制器转发方法。一种是工程实施方法，一种需要开发。

问题解决过程中不足之处

如果已经熟练使用ovs-appctl ofproto/trace产生数据包，就不用再花费更多时间去配置交换机验证了。
